PHASE 1 OPENWEBUI GPT-5.1 IMPLEMENTATION PROMPT PACK
===================================================

Document version: 1.0
Target environment: Windows PowerShell, Python 3.12, one cumulative repository
Phase boundary: Module 01 is already working; this pack builds Modules 02-05 only

PURPOSE
-------

This document is a manual implementation workbook for a developer who has
OpenWebUI with GPT-5.1 but does not have an AI coding agent or repository-aware
tools on the work computer. The prompts are deliberately repetitive. Each one
contains the interfaces GPT-5.1 needs for that bounded task so it does not need
to see the whole repository and does not silently invent a different contract.

Phase 1 ends at a deterministic, read-only hardening plan:

    real EZ-Scan job
      -> recorded results.zip
      -> safely selected results.xlsx
      -> normalized findings
      -> SBOM and service classification
      -> plan.json and plan-summary.md

Phase 1 DOES NOT modify application source. Git worktrees, dependency mutation,
LangGraph, approvals, model calls, source repair, image rebuilding, rescanning,
PR creation, and Jenkins integration are later phases.


HOW TO USE THIS PROMPT PACK
---------------------------

1. Finish Module 01 first. Its offline tests and one approved live work-network
   smoke test must pass, including a valid results.zip download.
2. Process prompts in identifier order. Do not skip a prompt or begin the next
   module before the current module review prompt returns an accepted handoff.
3. Start a fresh OpenWebUI chat for every prompt. Copy everything between that
   prompt's START and END delimiters into the chat.
4. Replace every PASTE CURRENT FILE HERE placeholder with the complete current
   file. Replace every PASTE HANDOFF HERE and PASTE TEST OUTPUT HERE placeholder
   with literal, unedited content.
5. GPT-5.1 must return complete files, not diffs. Save each returned file at the
   exact path printed above its code block. Never combine a partial response
   with an older file.
6. Run the prompt's PowerShell validation commands yourself. GPT-5.1 has no
   terminal and must not claim that it ran them.
7. If validation fails, use the module review prompt or the reusable failure
   diagnosis prompt near the end of this document. Paste the entire error,
   including the command and traceback.
8. Retain each accepted MODULE_HANDOFF block. It is the compact, authoritative
   context to carry into the next prompt.
9. Do not place real routes, registry names, email addresses, credentials, or
   returned EZ-Scan tokens into OpenWebUI. Use sanitized placeholders.

PROMPT RESPONSE CONTRACT
------------------------

Every generation prompt in this pack tells GPT-5.1 to follow these rules:

- Treat the supplied frozen interface ledger as authoritative.
- If a required current file or handoff is missing, respond only with:

      MISSING_INPUTS
      - <exact missing item>

- Never guess an existing file's contents or rename a frozen public symbol.
- Return the complete content of every ADD or REPLACE FROM PASTED CONTENT file.
- Put the exact repository-relative path before each fenced code block.
- Do not return patches, unified diffs, ellipses, TODOs, placeholders,
  pseudocode, omitted imports, or comments such as "existing code here."
- Limit one response to one large file or two tightly coupled small files.
- If the requested output cannot fit, return MISSING_INPUTS with
  "response must be split" instead of truncating a code block.
- Do not claim tests were executed. Provide commands and state that the human
  must run them.
- Preserve Python 3.12, Pydantic 2, strict validation, deterministic output,
  offline tests, and the secret-handling rules in this document.

---------------------------

```mermaid

    SAFE["Sanitized run metadata"]

    ENV --> API
    API --> TOKEN
    TOKEN -->|Authorize header only| API
    API --> SAFE --> ART
```

Trust rules:

- The per-job access token is never stored in .env, JSON, YAML, Markdown,
  fixture files, logs, tracebacks, model representations, or console output.
- Email is an API input but is not persisted in Phase 1 run records.
- Internal URLs, image names, and registry authorities must be supplied through
  the work machine's environment or local configuration, never hardcoded.
- Normal tests use fakes and local fixtures. A live scan requires an explicit
  opt-in environment variable.

Wire invariants:

artifact paths cannot be absolute, contain parent escapes, or resolve outside
the run directory.

Coordinator API:

    class EzScanLifecycleCoordinator:
        def __init__(
            self,
            backend: EzScanBackend,
            artifact_store: ArtifactStore,
        ) -> None

        def run(
            self,
            request: EzScanSubmissionRequest,
        ) -> EzScanSubmissionRecord

The run method is one process: create/get the run, submit, persist sanitized
submission metadata, observe initial status, wait if needed, download only on
passed status, persist numbered status records, update manifest, and return the
final sanitized record. It never serializes the submission object because that
object contains the required secret token.

Archive invariants:

- Reject an outer archive larger than max_archive_bytes.
- Reject more than max_members and expanded totals over max_expanded_bytes.
- Reject absolute paths, drive prefixes, parent traversal, backslashes,
  symlinks, encrypted members, and unsafe names.
- Ignore directory entries and __MACOSX metadata.
- Select exactly one .xlsx member; zero or multiple candidates fail closed.
- Extract only that workbook to an atomic temporary path beneath the requested
  destination directory.
- Enforce max_workbook_bytes while streaming and verify SHA-256.
- Inspect the XLSX as a ZIP and reject any vbaProject.bin entry.

Workbook invariants:

- Open with openpyxl load_workbook(read_only=True, data_only=True).
- The first non-empty row is the header.
- Required canonical columns are cve, package, installed_version,
  fixed_version, and severity.
- Blank, N/A, NA, none, and - fixed versions normalize to None.
- Unknown severity normalizes to UNKNOWN.
- Known prefixes such as python:name infer an ecosystem when no explicit
  package-type column exists.
- Deduplicate on cve, package, ecosystem, installed_version, fixed_version,
  preserving the first sheet and row evidence.
- finding_id is cve:ecosystem-value:package.
- Threshold order is LOW < MEDIUM < HIGH < CRITICAL. UNKNOWN is excluded from
  threshold filtering unless explicitly selected in severities.

Classification order:

1. Map service by finding.service, finding.image, SBOM image, then a single
   service fallback. Multiple services with no match remain unresolved.
2. Match SBOM by package name, installed version, and known ecosystem. If the
   finding ecosystem is unknown, match only name and version.
3. A known finding ecosystem that conflicts with a known SBOM ecosystem is an
   explicit package_type_mismatch.
